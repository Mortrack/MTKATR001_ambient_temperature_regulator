Detailed Description

This Class provides several methods to enable the application with the capability of communicating and sending some desired Payload Data through the use of the ETX OTA Communication Protocol over the UART Hardware Protocol.

The way that this Class sends Payload Data to a desired slave device is by calling the PcTool UART API made by Mortrack (cmira.nosp@m.ndam.nosp@m.eza3@.nosp@m.hotm.nosp@m.ail.c.nosp@m.om) and, therefore, leaving all the heavy processing of data and ETX OTA Protocol interaction to that API, which is to report back by sending a corresponding ETX OTA Status Code as stated in com.mortrack.hostbleapp.etxotaprotocol.EtxOtaProtocolFactory.EtxOtaStatus in order to be able to know the result of the processes made by the PcTool UART API.

This method interacts with the PcTool UART API for sending the required information so that the mentioned API gets to make all the Hardware Protocol Configurations, Data Processing and Data Handling instead of having it done on this Java Project side.

Local com.mortrack.hostbleapp.utils.TerminalProcess Type variable terminalProcess: Used to hold a Terminal Process Object, which will be later used for communicating with the PcTool UART API made by Mortrack and to request to it to send a desired Payload under the requested Configuration Settings.

Local String Type variable ret: Used to hold the String value of the interpreted integral number of whatever exception code value is returned by the PcTool UART API made by Mortrack. Note that the returned values will stand for the com.mortrack.hostbleapp.etxotaprotocol.EtxOtaProtocolFactory.EtxOtaStatus Exception Codes but in their respective interpreted integral number, which is to later be converted to its corresponding integer value via the com.mortrack.hostbleapp.etxotaprotocol.UartEtxOtaProtocol.etxOtaStatusHandler function.

Local String Type variable command: Used to populate it with the Terminal Command required for executing the PcTool UART API made by Mortrack and to give to it the desired Configuration Parameters and also to request to it to send a desired Payload under those Configuration Settings.

Delay in microseconds that it is to be requested to apply each time before calling the RS232_PollComport() function of the Teuniz RS232 Library that is used in the PcTool API.

For all the calls to the RS232_PollComport() function previously mentioned, this delay is called once, except in the send_etx_ota_data() function of the PcTool API, where this delay is called twice.

Note
The Teuniz RS232 Library suggests to place an interval of 100 milliseconds, but it did not worked for me that way. Instead, it worked for me with 500ms.

Delay in microseconds that it is to be requested to apply in case that starting an ETX OTA Transaction fails once only.

Note
The slave device sometimes does not get the start of an ETX OTA Transaction after its UART Timeout expires, which is expected since there is some code in the loop that the slave device has there that makes it do something else before waiting again for an ETX OTA Transaction, but that should be evaded by making a second attempt with the delay established in this variable.
