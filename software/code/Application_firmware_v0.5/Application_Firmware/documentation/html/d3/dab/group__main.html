Detailed Description

This is the Main Module, where the main part of the application of the whole code lies.

The purpose of this module is to provide an Application Firmware that already provides the code to manage and handle Firmware Updates via the Mortrack's ETX OTA Protocol which uses the Firmware Update Configurations sub-module .

Note
The Main module template is actually functioning as an Application Firmware that has a Bootloader counterpart and that should be located in an independent project to this, next to the root folder where the Main module project's File Path is located at.

In addition, the ETX OTA Protocol also optionally allows to receive Custom Data from the host machine for whatever purpose the implementer requires it for, which will be used here for updating some parameters of the MTKATR001 System (for more details, see the etx_ota_status_resp_handler function of the Main module program).

Note
In the case of receiving an ETX OTA Custom Data Request from the host machine, then the received data will be handled via the etx_ota_custom_data struct at the etx_ota_status_resp_handler function in order to be able to immediately process any new received ETX OTA Custom Data from the host.

The logic contemplated in this Main module template assumes that neither Bootloader or Application Firmware Updates will be applied by this Application Firmware. Instead, whenever receiving those requests, our MCU/MPU should apply a Software Reset in order to enter into Bootloader mode again to try receiving those Firmware Update requests there, since it is expected that only the Bootloader installs Firmware Images into our MCU/MPU.

Note
To learn more details on how to update either the Bootloader or Application Firmwares of this project, feel free to review the Doxygen documentation available at the main.c file of the Bootloader Firmware project.

As for the actual purpose of the application code of this program, the Main module has the goal of making the MTKATR001 Device to regulate a desired/controlled internal ambient temperature. The user should be able to update the MTKATR001 Device with customized values for the desired internal ambient temperature.

Application examples of the MTKATR001 Device would be, at least for the author of this project, to measure and characterize electrical or physical components under controlled internal ambient temperature with the intention of developing mathematical models that can reliably predict their behaviour. This could help to better understand certain scientific behaviors or phenomena, or could also be used for other engineering designs.

Note
WARNING: Whenever the peripheral configurations of this project are updated via the STM32CubeMx App, update those automated changed manually with the USER CODE BEGIN Private defines of the main.h file of this Main module , where it will be responsability of the person that updated this project to keep and update the corresponding Doxygen documentation made in that main.h file. The reason this has to be made is because the USER CODE BEGIN Private defines made available there by the author of this project are actually the ones that were autogenerated by the STM32CubeMx App, but that the author moved there in order to add them some Doxygen Documentation and to prevent the STM32CubeMx App to erase them whenever that App is used to update the Peripheral configurations of this Main module .

◆ Cold_Water_Pump_GPIO_Output_GPIO_Port

#define Cold_Water_Pump_GPIO_Output_GPIO_Port   GPIOA

Label for Pin PA8 in Output Mode, which controls the Cold Water Pump of our MCU that the Main module module will use in its program for turning On or Off the Cold Water Pump to assist the Cold Fan to respectively throw in Cold Air or not inside the MTKATR001 System.

The following are the output states to be taken into account:

* 0 (i.e., Low State and also Cold Water Pump turned Off) = The Cold Water Pump has been turned Off; no cold water will circulate around the outer side of the Cold Fan.
* 1 (i.e., High State and also Cold Water Pump turned On) = The Cold Water Pump has been turned On; cold water will circulate around the outer side of the Cold Fan.

◆ etx_ota_status_resp_handler()

void etx_ota_status_resp_handler ( ETX_OTA_Status  resp)

ETX OTA Status Response Callback.

This function is called by the ETX OTA Protocol module each time an ETX OTA Transaction has been completed, so that the implementer of ETX OTA Protocol module can customize handling the resulting ETX OTA Status Exception Code.

Note
For more details on how this function works with respect to the ETX OTA Protocol, see the Doxygen Documentation available for it at the ETX OTA Protocol module .

As a summary, this function will handle any ETX OTA Firmware Update Requests and, with respect to the MTKATR001 System, this function will handle any ETX OTA Custom Data Transactions, which are expected to contain the data to update some Global Variables of the Main module in the following way:

  1. desired_internal_ambient_temperature = 1st byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 42
  2. desired_hot_fan_duty_cycle = (2nd byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data) + (3rd byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data)
  3. desired_cold_fan_duty_cycle = (4th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data) + (5th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data)
  4. desired_hot_water_temperature = 6th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data
  5. desired_hot_water_min_temperature = 7th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data
  6. desired_cold_water_max_temperature = 8th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 42

where desired_hot_fan_duty_cycle and desired_cold_fan_duty_cycle will be always set to a value of 100 whenever their retrieved values from the ETX OTA Custom Data Transaction are equal or greater than 100 respectively.

Note
The valid value range for each byte contained inside the etx_ota_custom_data_t::data field of etx_ota_custom_data is the following:
\(32 \leq\) valid value of each byte in the etx_ota_custom_data_t::data field \(\leq 126\)
where it is obviously expected that the values to be given to each byte in the etx_ota_custom_data_t::data field are expected to be whole numbers.
Parameters
respResulting ETX OTA Status Exception Code of the ETX OTA Transaction that has just been completed, where the only possible values that can be given are the following:
  • ETX_OTA_Status::ETX_OTA_EC_OK (ETX OTA Transactions continues in this case right before this callback function) In this case, some ETX OTA Custom Data has been received from the host.
  • ETX_OTA_Status::ETX_OTA_EC_STOP (ETX OTA Transactions are stopped in this case right before this callback function) In this case, the host has requested to stop a currently on-going ETX OTA Transaction.
  • ETX_OTA_Status::ETX_OTA_EC_NR (ETX OTA Transactions continues in this case right before this callback function) In this case, the host stopped responding to our MCU/MPU during an ETX OTA Transaction.
  • ETX_OTA_Status::ETX_OTA_EC_NA (ETX OTA Transactions are stopped in this case right before this callback function) In this case, an ETX OTA Firmware Update has been requested by the host (remember that this ETX OTA Protocol module cannot install Firmware Images and, therefore, a reboot is suggested to make the Bootloader Firmware of our MCU/MPU to install that Image).
  • ETX_OTA_Status::ETX_OTA_EC_ERR (ETX OTA Transactions are stopped in this case right before this callback function) In this case, an Error has happened during an on-going ETX OTA Transaction.

This global Variable will be used as a threshold so that whenever the Cold Water's Temperature is higher than this point, then the MTKATR001 System will emit a signal to the user to request to him/her to change the Cold Water for one colder than the value assigned to this variable.

uint8_t desired_hot_fan_duty_cycle = 100

Global variable that contains the Hot Fan Duty Cycle desired in the MTKATR001 System.

Note
The fan controlled by the PWM to which this duty cycle is linked to will bring hot air inside the MTKATR001 System.
This value should always be equal or greater and 0 and equal or lower than 100.

◆ desired_hot_water_min_temperature

uint8_t desired_hot_water_min_temperature = 50

Global variable that contains the Hot Water Minimum Temperature desired in the MTKATR001 System.

This Global Variable will be used as a threshold so that whenever the Hot Water's Temperature lowers below this point, then the MTKATR001 System will start heating it until it reaches the value of the desired_hot_water_temperature Global Variable.

◆ desired_hot_water_temperature

uint8_t desired_hot_water_temperature = 60

Global variable that contains the Hot Water Temperature desired in the MTKATR001 System.

This Global Variable will basically define the temperature at which the MTKATR001 System will regulate the Hot Water to, which is the Water that will be used to bring Hot Air inside the MTKATR001 System via the Hot Fan.
