Detailed Description

This is the Main Module, where the main part of the application of the whole code lies.

The purpose of this module is to provide a template code for any general purpose application that is desired to be developed for an STM32 MCU/MPU in which such a template consists of an Application Firmware that already provides the code to manage and handle Firmware Updates via the Mortrack's ETX OTA Protocol which uses the Firmware Update Configurations sub-module .

Note
The Main module template is actually functioning as an Application Firmware that has a Bootloader counterpart and that should be located in an independent project to this, next to the root folder where the Main module project's File Path is located at.

In addition, the ETX OTA Protocol also optionally allows to receive Custom Data from the host machine for whatever purpose the implementer requires it for.

In the case of receiving an ETX OTA Custom Data Request from the host machine, then the received data will be handled via the etx_ota_custom_data struct at the etx_ota_status_resp_handler function in order to be able to immediately process any new received ETX OTA Custom Data from the host.

Note
To learn more details about how to configure and handle ETX OTA Custom Data Requests, see the init_firmware_update_module and etx_ota_status_resp_handler functions.

The logic contemplated in this Main module template assumes that neither Bootloader or Application Firmware Updates will be applied by this Application Firmware. Instead, whenever receiving those requests, our MCU/MPU should apply a Software Reset in order to enter into Bootloader mode again to try receiving those Firmware Update requests there, since it is expected that only the Bootloader installs Firmware Images into our MCU/MPU.

◆ GPIO_is_hm10_default_settings_Pin

#define GPIO_is_hm10_default_settings_Pin   (GPIO_PIN_14)

Label for the GPIO Pin 14 towards which the GPIO Pin PC14 in Input Mode is at, which is used so that our MCU can know whether the user wants it to set the default configuration settings in the HM-10 BT Device or not.

The following are the possible values of this Pin:

* 0 (i.e., Low State) = Do not reset/change the configuration settings of the HM-10 BT Device.
* 1 (i.e., High State) = User requests to reset the configuration settings of the HM-10 BT Device to its default settings.

◆ GPIO_MCU_LED1_Pin

#define GPIO_MCU_LED1_Pin   (GPIO_PIN_13)

Label for Pin PC13 in Output Mode, which is the Green LED1 of our MCU that the Main module module will use in its program for indicating to the user whenever our MCU/MPU gets a software error or not.

The following are the output states to be taken into account:

* 0 (i.e., Low State and also LED1 turned On) = MCU got a Software Error.
* 1 (i.e., High State and also LED1 turned Off) = MCU has no software error.

◆ etx_ota_status_resp_handler()

void etx_ota_status_resp_handler ( ETX_OTA_Status  resp)

ETX OTA Status Response Callback.

This function is called by the ETX OTA Protocol module each time an ETX OTA Transaction has been completed, so that the implementer of ETX OTA Protocol module can customize handling the resulting ETX OTA Status Exception Code.

Note
For more details on how this function works with respect to the ETX OTA Protocol, see the Doxygen Documentation available for it at the ETX OTA Protocol module .

As a summary, this function will handle any ETX OTA Firmware Update Requests and, with respect to the MTKATR001 System, this function will handle any ETX OTA Custom Data Transactions, which are expected to contain the data to update some Global Variables of the Main module in the following way:

  1. desired_internal_ambient_temperature = 1st byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 42
  2. desired_hot_fan_duty_cycle = (2nd byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 32) + (3rd byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 32)
  3. desired_cold_fan_duty_cycle = (4th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 32) + (5th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 32)
  4. desired_hot_water_temperature = 6th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data
  5. desired_hot_water_min_temperature = 7th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data
  6. desired_cold_water_max_temperature = 8th byte of data from the etx_ota_custom_data_t::data field of etx_ota_custom_data - 42

where desired_hot_fan_duty_cycle and desired_cold_fan_duty_cycle will be always set to a value of 100 whenever their retrieved values from the ETX OTA Custom Data Transaction are equal or greater than 100 respectively.

Note
The valid value range for each byte contained inside the etx_ota_custom_data_t::data field of etx_ota_custom_data is the following:
\(34 \leq\) valid value of each byte in the etx_ota_custom_data_t::data field \(\leq 126\)
where it is obviously expected that the values to be given to each byte in the etx_ota_custom_data_t::data field are expected to be whole numbers.
Parameters
respResulting ETX OTA Status Exception Code of the ETX OTA Transaction that has just been completed, where the only possible values that can be given are the following:
  • ETX_OTA_Status::ETX_OTA_EC_OK (ETX OTA Transactions continues in this case right before this callback function) In this case, some ETX OTA Custom Data has been received from the host.
  • ETX_OTA_Status::ETX_OTA_EC_STOP (ETX OTA Transactions are stopped in this case right before this callback function) In this case, the host has requested to stop a currently on-going ETX OTA Transaction.
  • ETX_OTA_Status::ETX_OTA_EC_NR (ETX OTA Transactions continues in this case right before this callback function) In this case, the host stopped responding to our MCU/MPU during an ETX OTA Transaction.
  • ETX_OTA_Status::ETX_OTA_EC_NA (ETX OTA Transactions are stopped in this case right before this callback function) In this case, an ETX OTA Firmware Update has been requested by the host (remember that this ETX OTA Protocol module cannot install Firmware Images and, therefore, a reboot is suggested to make the Bootloader Firmware of our MCU/MPU to install that Image).
  • ETX_OTA_Status::ETX_OTA_EC_ERR (ETX OTA Transactions are stopped in this case right before this callback function) In this case, an Error has happened during an on-going ETX OTA Transaction.

◆ get_adc_output()

static HAL_StatusTypeDef get_adc_output ( ADC_HandleTypeDef *  hadc,
uint32_t  adc_channel,
float *  dst 
)
static

Selects a desired Channel at a specified ADC of our MCU/MPU and then gets the current output from it.

This function will start by changing the Configuration Settings of the given ADC Handler to the requested Channel. Then that ADC will be enabled to then Poll the Output of the requested ADC and Channel. Finally, the Output Value will be stored into the requested Destination and then this ADC will be stopped.

@notoe This function will always set a sampling time of 239.5 ADC Cycles for the Channel selected.

Parameters
[in]hadcPointer towards the structure of the ADC from which it is desired to select a desired Channel.
adc_channelChannel that wants to be selected from the desired ADC.
[out]dstPointer towards the Memory Address Location where the obtained ADC Output will be stored at.
Return values
HAL_OKIf there are no HAL Errors during the ADC and Channel selection and the retrieval of the respective output value from the requested ADC and Channel.
HAL_ERRORIf there was a HAL Error during the processes made by this function for the requested ADC and Channel. Note that if this case gives place, then the ADC's Output might not have been retrieved and, therefore, it would be best to consider that the Output was not obtain for safety purposes.
HAL_BUSYIf the ADC was Busy during a request made to it by this function. Note that if this case gives place, then the ADC's Output might not have been retrieved and, therefore, it would be best to consider that the Output was not obtain for safety purposes.

◆ validate_application_firmware()

static void validate_application_firmware ( )
static

Validates the CRC of our MCU/MPU's current Application Firmware and, only in the case that the Application Firmware is not valid or is corrupted, then this function will endlessly loop via a while() function and set the GPIO_MCU_LED1_Pin to GPIO_PIN_RESET state.

This function reads the data stored in the Firmware Update Configurations sub-module via the fw_config Global struct to get the recorded CRC of the Application Firmware and it calculates the CRC of the actual Application Firmware currently installed in our MCU/MPU. After this, both CRCs are used for validating our MCU/MPU's current Bootloader Firmware. If both CRCs match, then this function will terminate. Otherwise, it will endlessly loop via a while() function and set the GPIO_MCU_LED1_Pin to GPIO_PIN_RESET state.

Note
The fw_config Global struct must have already been populated with the latest data written into the Firmware Update Configurations sub-module before calling this function.
